# 개선점 종합 진단 프롬프트 (OMG Studios Manager)

> 재사용 프롬프트. 새 Claude Code 세션·서브에이전트에 **아래 "프롬프트" 코드펜스 전체**를
> 복붙해, 우선순위가 매겨진 개선점 리포트를 받는다. 슬래시 커맨드로도 등록됨 → `/audit`.

## 사용법
1. 새 세션을 열고 아래 코드펜스를 통째로 붙여넣는다(또는 `/audit` 실행).
2. 리포트가 **8개 축 + `파일:라인` 인용 + 심각도 분류**를 갖췄는지 확인한다.
3. Top 1~2 지적을 실제 코드에서 교차검증(할루시네이션 점검)한 뒤 착수한다.

## 프롬프트 (복붙)

```
당신은 OMG Studios Manager 코드베이스를 점검하는 시니어 코드 감사자입니다.
목표: 이 레포의 개선점을 "종합적으로" 찾아, 우선순위가 매겨진 리포트를 작성합니다.
코드는 수정하지 말고 진단 리포트만 작성하세요.

## 프로젝트 컨텍스트 (먼저 숙지 — 오탐 방지)
- 스택: Node ≥20, Express 4 (CommonJS), SQLite(better-sqlite3 운영 / node:sqlite 폴백),
  서버 렌더 HTML(src/views*.js) + 클래식 폼 POST + 최소 JS(public/js/app.js), Tailwind CLI.
- 인증: 전원 Google OAuth + 화이트리스트(users), 3단계 역할(owner/chief/staff), httpOnly 서명 JWT.
  미들웨어 requireAuth / requireEditor(대표 차단) / requireChief / requireInvoice.
- **내부 전용 도구** 전제: 로그인 직원이 전 프로젝트를 열람(클라이언트 범위 강제 없음)하는 것은
  의도된 설계다. 이를 "인가 누락"으로 오인하지 말 것.
- 의도적 설계 — 기계적으로 지적 금지(맥락 보고 판단):
  · better-sqlite3 **동기 API**는 소규모 내부도구라 의도적 → "동기 블로킹"을 무조건 지적하지 말 것.
  · CSP 인라인 스크립트 0 — 모든 JS는 public/js/app.js 외부 파일(인라인 핸들러 추가는 회귀).
  · DB CHECK 제약 미사용 — 코드 상수(config.js)가 enum 단일 진실원천(마이그레이션 지옥 회피).
  · 돈=정수(원), 날짜="YYYY-MM-DD" 문자열. 정적 자산 캐시버스팅(?v=).
- 이미 처리된 것 재지적 금지: CLAUDE.md의 "빠진 함정"·"주요 변경 이력"을 먼저 읽고,
  이미 해결된 항목(CSRF 기본거부·OAuth 논스·SSRF 차단·채번 원자화·세션 청구잠금·
  로고 매직바이트 등)은 다시 지적하지 말 것.

## 작업 방식
1. 먼저 레포 구조 파악: CLAUDE.md(설계 일지)·WORKFLOW.md·package.json·src/ 전체.
2. 아래 8개 축을 "하나도 빠짐없이" 점검. 각 축마다 점검을 시도했음을 보일 것
   (유의미한 발견이 없으면 "이 축에서 발견 없음"이라고 명시).
3. 모든 지적에 반드시 `파일경로:라인`을 인용. 인용 없는 지적 금지.
4. 추측 금지. 코드로 확인되지 않으면 "확인 필요"로 표시. 없는 문제를 지어내지 말 것.

## 점검 축 (체크리스트 — 각 축을 표제로)
1. 정확성/버그: 엣지케이스(빈값·음수·중복·null), async/콜백 오류, off-by-one, 깨진 제어흐름,
   잘못된 가정, 폼↔라우트 파라미터명 불일치.
2. 보안: 인증/인가 우회(미들웨어 빠진 라우트), SQL 인젝션(문자열 결합 쿼리), 입력 검증 누락,
   세션/시크릿/토큰 관리, XSS(esc 누락)·CSRF, 파일 업로드/경로 탈출.
3. 데이터 무결성: 트랜잭션 경계(다단계 쓰기), 동시 쓰기/경쟁, 스키마·멱등 마이그레이션,
   FK·CASCADE·SET NULL 정합, 삭제 가드, 스냅샷 일관성, 백업·복구 가정.
4. 성능: N+1 쿼리(반복문 안 쿼리), 누락된 인덱스, 큰 결과셋, 불필요한 재조회·렌더링.
5. 코드 품질/구조: 비대한 파일·함수(특히 src/data.js 1459줄·projects.routes.js 890줄),
   중복 로직(헬퍼로 묶을 것), 일관성 없는 네이밍, 죽은 코드, 모듈 분리 기회.
6. 에러 처리/관측성: 누락된 try/catch, 삼켜진 예외(빈 catch), 로깅 부족·과다,
   불친절한 사용자 에러 메시지, fail-safe 경로(알림·외부연동) 검증.
7. 테스트/CI: 자동화 테스트 전무(현재 test 스크립트·테스트 파일·CI·린트 모두 없음) —
   어디부터 테스트가 가장 가치 있는지(청구 채번·VAT·금액·세션 겹침·권한 매트릭스 등),
   eslint·prettier·GitHub Actions 부재의 위험.
8. UX/접근성 + 문서: UI 일관성(공통 헬퍼 미사용), 빈/로딩/에러 상태, 모바일(16px 입력),
   접근성(aria·대비·포커스), **문서(CLAUDE.md/WORKFLOW.md)와 코드 불일치**.

## 출력 형식
각 발견 항목:
- **[Critical / High / Medium / Low] 한 줄 요약**
  - 위치: `파일:라인`
  - 문제: 무엇이 왜 잘못/위험/비효율인지
  - 영향: 발생 시 구체적 시나리오
  - 제안: 구체적 수정 방향(가능하면 코드 스케치)
  - 확신도: 확실 / 확인 필요

마지막에:
- **요약 표**: 심각도별 개수
- **Top 5 우선 조치**: 영향 대비 수정비용이 좋은 순서로 즉시 착수 권장.

검증 팁(로컬 동작 확인이 필요하면): `pkill -f "src/server.js"` 후
`DEV_LOGIN=1 node src/server.js`, POST는 `-H "Origin: http://localhost:3000" -H "Sec-Fetch-Site: same-origin"`,
한글 쿼리스트링은 `--data-urlencode` 사용.
```
