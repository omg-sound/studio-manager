# 연락처 마스터-디테일 전환 — 설계

> 2026-07-17 · 사용자 요청("연락처를 애플 연락처식으로 바꾸자 — 폭 맞추기 피곤하다")
> 대상: `/contacts`(연락처 메뉴) · `/clients?group=associate`(클라이언트 › 관계자 탭)

## 배경 · 문제

연락처 목록은 `contactTable`(이름·소속·직함·역할·전화·이메일 6열 표)이다. 열 폭이 반복 이슈였다:
이름을 좁히면 활동명 병기 라벨(`Kim George Han (김조한)`=183px)이 잘리고, 회사명(`언디파인드엔터테인먼트주식회사`=208px)까지
보이려면 소속에 240px가 필요하고, 역할 배지(최장 3개=195px)까지 한 줄에 넣으면 남는 폭이 없다.
2026-07-16~17에만 폭 조정이 세 번 있었다(11rem→13rem, 역할 hide md→xl→lg, 반응형 폭 추가).

근본 원인은 폭 계산이 아니라 **화면의 성격 오인**이다. 표는 여러 행을 나란히 비교·스캔할 때 값어치가 있는데
(청구 목록의 금액·상태처럼), 연락처에서 실제로 하는 일은 "이 사람 찾아서 전화·이메일 확인"이다.
사용자 확인: **목록에서 열지 않고 전화·이메일을 복사하는 흐름은 거의 안 쓴다 — 다 열어서 확인한다.**

→ 이름 한 열(마스터) + 상세 패널(디테일)로 바꾸면 비교용 열이 사라지므로 **폭 문제 자체가 소멸**한다.

## 목표 · 비목표

**목표**
- 연락처·관계자를 마스터-디테일(왼쪽 이름 목록 + 오른쪽 상세)로 전환
- 열 폭 튜닝이 필요한 지점을 이 두 화면에서 제거
- 목록 상한 없이 전 명단 노출

**비목표(이번에 안 함)**
- 클라이언트 상세(업체·아티스트·그룹)의 인라인 편집 방식 변경 — 그대로 둔다
- 클라이언트 업체/아티스트/그룹 탭의 표 — 그대로 둔다(비교·스캔 성격)
- 애플식 ㄱㄴㄷ 인덱스 레일 — 실시간 필터가 있어 불필요(YAGNI)
- 오른쪽 패널의 JS 부분 교체(SPA화) — 서버 렌더 유지

## 확정된 결정(사용자)

| 항목 | 결정 |
|---|---|
| 목록에서 직접 복사 | 거의 안 씀 → 이름 한 열로 축소해도 무방 |
| 적용 범위 | 연락처 메뉴 + 클라이언트 관계자 탭 **둘 다** |
| 왼쪽 목록 행 | **이름만**(소속·역할 병기 없음) |
| 오른쪽 패널 | **애플식 읽기 뷰 + [편집] 버튼**(연락처 한정으로 "상세=바로 편집" 뒤집기) |
| 읽기 뷰 구성 | **탭 없이 한 화면 스크롤** |
| 좁은 화면 | 목록만 → 이름 누르면 상세 전체 화면(아이폰 연락처식) |
| 목록 상한 | **제거** — 전 명단 노출, 부담되면 그때 대응 |

## 접근 방식 — A안(채택)

`/contacts/:id` **자체를 2단 페이지로 만든다.** 새 URL 개념을 만들지 않고 기존 상세 경로의 렌더링만 바꾼다.

- 서버 렌더 그대로 → JS 0, CSP 무관, 뒤로가기·북마크·딥링크 공짜
- 선택 시 전체 페이지 로드 — 앱 전체가 이미 그 방식이라 이질감 없음
- `?return=`으로 목록 상태를 실어 나르던 장치가 불필요해짐(왼쪽에 목록이 늘 있음)

**기각한 안**
- B안 `?sel=` 쿼리: 상세 URL이 `/contacts/:id`와 두 벌이 됨. 이득 없음
- C안 JS로 오른쪽만 fetch 교체: History API 직접 관리 + JS 없으면 죽음. "클래식 폼 POST + 최소 JS" 원칙 위반

> 예외: 관계자 탭은 경로가 `/clients?group=associate`(탭 쿼리 기반)이라 선택만 `?sel=<id>`로 받는다.
> 연락처 메뉴는 경로 파라미터, 관계자 탭은 쿼리 — 각 라우트에서 자연스러운 쪽을 쓴다.

## 화면 · 라우트

### 1. `/contacts` — 목록 + 빈 패널
- 왼쪽: 3탭(외부 연락처/외주 작업자/녹음실 스태프) + 검색 + **전 명단 이름 목록**
- 오른쪽: `emptyState("연락처를 선택하세요")`
- 좁은 화면(<1024): 왼쪽만

### 2. `/contacts/:id` — 목록(선택 강조) + 읽기 뷰
읽기 뷰는 탭 없이 위에서부터:

1. **헤더** — 이름(`personName` = 본명 호칭 (활동명), 큰 글씨) + 역할 배지(`classifyParty`), 우상단 **[편집]**
2. **연락 정보** — 전화 · 이메일 · 현금영수증(각 `copyable`)
3. **소속** — 현재 회사·직책·부서 + 소속 이력 타임라인(현재/종료 배지, **읽기 전용**)
4. **메모**
5. **참여 내역** — 프로젝트 표(아티스트·제작사·프로젝트·작성일) + 세션 표(날짜·시간·종류·프로젝트·상태) — 2026-07-17 만든 `dataTable` 렌더 재사용
6. **연동 정보** — 대표 클라이언트 · 아티스트로 보기 · 담당자 연동 배지(현행 `derivedBits`)

좁은 화면(<1024): 오른쪽만 + '← 연락처' 뒤로가기

### 3. `/contacts/:id/edit` — 목록 유지 + 오른쪽만 편집 폼
- 지금은 상세로 리다이렉트만 하는 경로를 **편집 모드로 되살린다**
- 내용 = 현행 '상세 정보' 탭 그대로: `contactForm`(dirty 저장) + 소속 이력 인라인 편집 + 소속 추가/이직 폼 + 삭제
- 저장·삭제 후 → `/contacts/:id` 읽기 뷰(삭제는 `/contacts`)
- 취소 → `/contacts/:id`

### 4. `/clients?group=associate[&sel=<id>]` — 관계자 탭 2단
- 왼쪽: 관계자 전 명단(이름만) — 클라이언트 탭바(업체/관계자/아티스트/그룹)는 그 위에 유지
- 오른쪽: `sel` 없으면 안내, 있으면 **같은 읽기 뷰**
- [편집]은 `/contacts/:id/edit?return=<현재 관계자 탭 주소>`로 이동(편집 폼을 두 곳에 두지 않는다).
  저장 후 `safePath(return)`으로 관계자 탭 복귀 — 백링크 규약(CLAUDE.md)과 동일
- 좁은 화면: 연락처 메뉴와 **같은 규칙**(`sel` 없으면 목록만, 있으면 읽기 뷰만 + '← 관계자' 뒤로가기).
  행 링크는 폭과 무관하게 항상 `?sel=<id>` 하나다 — 서버는 href를 하나만 렌더하므로 뷰포트별로 목적지를 달리할 수 없다

## 컴포넌트 경계

새 렌더러는 `src/views.contacts.js`(신설)에 모은다. `views.js`는 이미 크고, 연락처 전용 뷰는 독립 단위로 떼는 게 맞다.

| 단위 | 책임 | 입력 | 쓰는 곳 |
|---|---|---|---|
| `contactPanes({ left, right })` | 2단 레이아웃 골격 + 반응형 분기 | 좌/우 HTML | 연락처·관계자 |
| `contactNameList({ rows, selectedId, hrefFn, filterList })` | 이름만 목록(선택 강조·실시간 필터 마커) | 사람 배열 | 연락처·관계자 |
| `contactReadView(party, { affs, projects, sessions, editHref, extras })` | 읽기 뷰(6개 섹션) | 조회된 데이터 | 연락처·관계자 |
| `contactEditPane(...)` | 편집 폼 + 소속 이력 편집(현행 코드 이동) | 〃 | 연락처 |

- 데이터 조회는 지금처럼 라우트가 한다(뷰는 순수 렌더).
- `contactReadView`는 `editHref`를 받아 [편집] 목적지를 호출부가 정한다 → 관계자 탭은 연락처 메뉴로 보낼 수 있다.

## 반응형 규칙

- 골격: `lg` 기준 2단(`lg:grid lg:grid-cols-[18rem_minmax(0,1fr)]`), 미만은 한 단
- 목록 URL(선택 없음): 왼쪽 `block`, 오른쪽 `hidden lg:block`
- 상세 URL(선택 있음): 왼쪽 `hidden lg:block`, 오른쪽 `block`
- 서버가 선택 여부를 알고 클래스를 정하므로 **JS 없음**
- 페이지는 `layout({ wide: true })`(넓은 목록·표와 동일 취급)

## 데이터 · 규모(실측 2026-07-17, 프로덕션)

| 목록 | 건수 | 현재(6열 표) HTML |
|---|---|---|
| 외부 연락처 | 202 | 177KB |
| 관계자 | 151 | 127KB |
| 외주 작업자 | 4 | 19KB |
| 녹음실 스태프 | 5 | 19KB |

이름만 렌더하면 행당 HTML이 표 행의 1/4 수준 → 202건 전체 + 상세를 합쳐도 수십 KB. 상한 없이 안전.
**부담 임계**: 명단이 2,000건대로 늘어 목록 HTML이 수백 KB에 이르면 그때 가상 스크롤·서버 페이징을 검토한다(사용자 결정: 지금은 안 함).

## 오류 · 예외

- `/contacts/:id`에 없는 id → 현행처럼 404 `errorPage`(2단 아님)
- 관계자 탭 `?sel=<없는 id>` 또는 관계자가 아닌 id → 오른쪽은 안내 문구, 목록은 정상(404 아님 — 탭 자체는 유효)
- 빈 목록(탭별) → 왼쪽에 현행 `emptyState`(스태프/외주/외부별 문구·CTA) 유지
- 삭제된 연락처의 상세를 열어둔 채 저장 → 현행 라우트 동작 유지
