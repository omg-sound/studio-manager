> ## 📛 표준 호칭 = "ERP" (2026-07-17 사용자 지시 — 항상 적용)
> **이 제품을 부르는 표준 이름은 'ERP'다.** 이전에 '앱·페이지·스튜디오 매니저·(사내) 웹앱·관리 시스템' 등으로 부르던 것을 모두 **'ERP'로 통일**한다. 사용자가 "ERP"라고 하면 곧 **이 제품(OMG Studios ERP)**을 가리키는 것으로 알아듣는다. 문서·사용자 노출 UI 서술어(제품명·메일·로그인 부제 등)는 'ERP'로 칭한다. **단 브랜드 워드마크 "OMG Studios"는 유지**(사이드바 로고·페이지 타이틀 `· OMG Studios`·메일 발신명). 인프라 식별자(`render.yaml` 서비스명 `omg-studios-manager`·`package.json` `name`)는 배포 안정성상 불변. 내부 코드 주석에서 실행체를 가리키는 일반적 "앱"은 그대로 둔다(제품 호칭이 아니라 러닝 인스턴스 지칭).
>
> ## 🎯 제품 정체성 (2026-07-10 확정 — 운영 원칙)
> **녹음 스튜디오 전용 버티컬(도메인 특화) 미니 ERP.** '웹앱'은 만든 형태일 뿐이고, 정체성은 **예약(수주)부터 작업·청구·정산까지 order-to-cash 흐름을 단일 데이터 모델(당사자 `parties`)로 통합한 사내 시스템**이다. 통합 모듈 = **운영**(세션·룸·프로젝트·작업)·**회계/재무**(청구·VAT·거래명세서·매출·미수)·**CRM**(클라이언트·연락처·소속 이력·구글 동기화)·**인사/정산 일부**(외주 정산·원천징수·하우스 계정). 스케줄러 하나·인보이스 툴 하나가 아니라 **이 흐름을 하나로 잇는 것**이 정체성이다.
> - **의도적으로 안 지는 무게(경계)**: 총계정원장(GL)·재고/구매·정식 급여·근태·멀티 조직/테넌트 = **범위 밖**. '가장 가벼운 경로'·삭제 중심 관리 철학과 일치하는 **경량 버티컬 ERP**를 유지한다(대형 ERP를 흉내 내지 않는다).
> - **스코프 판단 기준(신규 기능 검토 시 항상)**: *"이게 이 스튜디오의 운영–정산 통합 흐름에 속하나?"* — 속하면 넣고(그 흐름의 한 단계이거나 단일 진실원천을 강화하면 ○), 범용 ERP 기능을 흉내 내는 것이면 뺀다. 정체성 확장이 필요하면 코드보다 이 정의를 먼저 갱신·합의한다.

## 작업 원칙 — 카파시 가이드라인 (2026-07-10 사용자 지시, 항상 적용)

> 이 프로젝트의 모든 작업(기능·수정·점검)에 다음 4원칙을 적용한다. 신중 > 속도 편향(사소한 작업은 재량).
1. **코딩 전 생각**: 가정을 명시하고, 해석이 여러 개면 조용히 하나를 고르지 말고 제시·확인. 더 단순한 접근이 있으면 구현 전에 말한다. 불명확하면 멈추고 묻는다.
2. **단순함 우선**: 요청을 푸는 최소 코드만. 요청 밖 기능·1회용 코드의 추상화·요청 없는 유연성/설정·불가능한 시나리오의 에러 처리 금지. "시니어가 과하다고 할까?" 싶으면 단순화.
3. **수술적 변경**: 요청과 무관한 인접 코드·주석·포맷을 '개선'하지 않는다. 기존 스타일을 따른다. 무관한 죽은 코드는 지우지 말고 보고만. 내 변경이 만든 고아(import·변수)는 정리. **모든 변경 라인이 요청으로 소급돼야 한다.**
4. **목표 주도 실행**: 작업을 검증 가능한 기준으로 바꿔 완료를 증명한다(버그=재현 테스트 먼저→통과, 리팩터=전후 테스트 통과). 다단계 작업은 단계별 검증 계획을 먼저 세운다. — 기존 관행(가드레일 승격 #21·회귀 테스트·E2E 검증)과 같은 방향.

> **상황별 규율(2026-07-10 사용자 승인 — superpowers 스킬 선별 도입)**: **버그 수정** = `systematic-debugging`(수정안보다 근본 원인 먼저) + 재현 테스트 먼저 / **완료 주장 전** = `verification-before-completion`(검증 명령 실행·증거 확인 후에만 "완료") / **대규모 컷오버** = git worktree 격리(`using-git-worktrees` — main 커밋=자동배포라 미완 상태 커밋 금지). 그 외 superpowers 스킬(brainstorming·plans류·review류)은 OMC/내장 기능과 중복이라 미사용.

### 프로젝트
- **유형 구분 없음**: 세션 일정 자체가 클라이언트 방문·예약이고 녹음/믹스는 그 안에서 일어나므로 프로젝트 단에서 세션/작업을 나누지 않는다. **모든 프로젝트가 세션 일정 + 곡·콘텐츠를 동일하게** 갖는다. "+ 새 프로젝트"는 단일 진입. `project_type`/`PROJECT_TYPES`는 레거시 컬럼·상수로만 보존(신규=`session` 고정, 편집 시 기존값 유지, UI 미노출·드롭다운/배지 제거).
- 상세는 **탭**: `정보(메타) / 세션 일정 / 곡·콘텐츠 / 자료 전달 / 청구`(청구 탭=치프·대표·스태프. 첫 탭 라벨='정보', URL 키는 `?tab=project`[하위 호환]). 세션 일정 탭은 **전 프로젝트 노출**. 메타 편집은 **'정보' 탭**(첫 탭·기본), URL `?tab=`. **메타 폼 하단에 '프로젝트 삭제' 옆 '취소' 버튼**(저장하지 않고 프로젝트 목록으로, `dirtyActionRow({deleteFormId, cancelHref:"/projects"})` — 삭제·취소 나란히). `data-no-guard`로 dirty 네비게이션 가드의 커스텀 모달을 우회하되, 그것만으로는 브라우저 네이티브 beforeunload "나가시겠습니까?" 프롬프트를 못 막으므로 app.js 네비게이션 가드 클릭 핸들러가 `data-no-guard` 링크에서 **`bypass=true`(1초 타이머로 자동 해제)도 함께 세팅**해 beforeunload까지 통과한다(함정 #24).

- **겹침 경고 + 확인 후 강행**: 같은 룸·같은 시간 겹침을 **막지 않고 확인 후 등록 허용**. **모든 세션 종류가 룸을 점유**(`findSessionConflict`/`busySessionSlots` session_type 무관). **시작 시간 그리드는 폐지** — busy 슬롯은 겹침 경고 계산에만 쓴다. ①선택한 시작+소요가 겹치면 **시작 시간 아래 경고 문구**(`[data-conflict-warn]`, `overlapDetected` 클라이언트 근사), ②**제출 시 `confirm("이미 예약된 일정이 있습니다. 그래도 등록할까요?")`** → 승인하면 hidden `override_conflict=1` 전송 → 서버 `assertNoSessionConflict(f, excludeId, allowConflict)`가 겹침 검사를 건너뛴다(`conflictOverride`가 `input.override_conflict==="1"` 판정, create·update 공통). override 없이 겹치면 서버가 여전히 409(`sessionConflictMessage`) — JS 미동작·경합 대비 안전망. 구글 FreeBusy 하드차단은 **비활성화**(단일 캘린더로는 룸 구분 불가, 캘린더 일정 자동 생성/수정/삭제 동기화는 유지).

- **`/workers` 권한 분리**: 목록·상세 열람 + **정산(지급 처리/취소)** = 대표·치프(`requireInvoice`), 작업자 **추가·삭제·정보수정**(마스터) = 치프(`requireChief`). 추가 폼·삭제·수정은 치프에게만 노출. 외주 **정보 수정**(이름·전화·이메일·정산 정보)=상세 `POST /workers/:id/edit`(이름 변경 시 `track_tasks.engineer_name` 동기화로 정산 매칭 유지). 외주 **지급단가**(`worker_rate`) 입력은 작업 편집(`requireEditor`=치프·스태프)이라 대표는 단가는 안 건드리고 **지급만** 실행. 흐름: 치프/스태프 외주단가 입력 → 완료 → 대표/치프 정산 탭 지급. **목록 카드 밑 지급 요약 + 일괄 지급 처리**: 각 작업자 카드 바로 아래 미지급 건수·합계 한 줄(`listTasksForWorker` 재사용) + 오른쪽 끝 **[지급 처리]** 버튼(`POST /workers/:id/payout-all` — 그 작업자의 미지급 작업 전부를 한 번에 지급 처리, 대표·치프=`requireInvoice`). 미지급 없으면 요약 줄 자체가 안 보임. **미지급 항목 미리보기 한 줄**: 요약 줄 아래 `프로젝트명 · 작업종류/세션종류 날짜` 라벨을 콤마로 나열, 최대 3개+`외 N건`(작업+세션 통합 목록, `taskTypeLabel`/`formatYmdShort`, 라벨 맨 앞 `project_title` — 예 `루나 1집 - 타이틀곡 '월광' · 마스터링 7월 6일`). **정산 항목 식별**: `listTasksForWorker`가 트랙/프로젝트 조인에 `track_artist`(트랙 아티스트, 없으면 프로젝트 아티스트 폴백)를 조회, 목록 미리보기·상세 정산/참여 내역 작업 행 공용 `taskMeta`가 **아티스트 + 작업일(생성일)** 을 함께 표시(`보컬튠 · 곡아티스트 · 프로젝트/트랙 · 7월 1일`; 세션 항목은 `session_date`). **정산 정보(주민등록번호/사업자등록번호·계좌)**: 정보 수정 폼에 `id_number`(주민등록번호 또는 사업자등록번호 — 어느 쪽이든 한 칸에)·`bank_name`(은행)·`account_number`(계좌번호)·`account_holder`(입금자명/예금주). **`id_number`·`account_number`는 `db.encrypt`/`decrypt`로 암호화 저장**(민감정보/재무정보라 DB 파일 유출 시에도 평문 노출 방지, AES-256-GCM). 편집 폼은 치프가 볼 때 **복호화해 그대로 표시**(마스킹 없음 — 저장은 암호화, 표시는 인증된 치프에게만 평문). `bank_name`/`account_holder`는 평문 저장. **첨부 서류(주민등록증 사본·통장사본)**: `worker_files` 테이블(client_files와 동일 구조, kind별 1개 교체식) + 치프 전용 업로드·인증 다운로드·삭제(`POST/GET /workers/:id/files/:kind[...]`, 매직바이트 검증 PNG/JPEG/PDF, 공개 링크 없음). **담당자 목록 정렬 = 하우스 먼저·외주 나중**: `listProjectManagers`(전 앱 공용 — 세션·작업 엔지니어 select·예약담당자 select·`/workers`·`/settings` 담당자 탭이 전부 이 함수 하나를 씀) `ORDER BY active DESC, (user_id IS NULL) ASC, name COLLATE NOCASE`(활성 우선 → 하우스[user_id 있음] 먼저 → 외주 나중 → 그룹 내 가나다순). 클라이언트 첨부는 사업자등록증만(통장사본 폐기)이지만 **외주는 스튜디오가 돈을 지급하는 방향이라 통장사본이 필요**(client_files와 다른 이유). `storage-migrate.js`의 로컬→Drive 이관 대상(`LOCAL_TABLES`)에도 포함. **상세 3탭**: 기본 정보(기본 탭 — 정보 수정 펼침 카드+필드 라벨+첨부 서류, 비치프는 읽기 카드) / 참여 내역 / 정산. **드롭존 붙여넣기 단축키 OS 구분**(맥=⌘V·그 외 Ctrl+V — 서버는 Ctrl+V 렌더, app.js가 맥에서 라벨·aria 치환). **뷰어 크기 규칙**(viewer.js — 원본 이상 업스케일 금지·상한 높이 85%×폭 60%·하한 360px·리사이즈 1회 재시도). **정산 탭**: ①**이체 정보 카드**(정산 탭 상단 — 은행·계좌번호[복호화·클릭 복사]·예금주, 미입력 시 경고 배지) ②**지급월별 이력**(지급완료를 worker_paid_date 월로 그룹, 월 헤더에 지급액·원천세·실지급 — 원천세 신고[지급월 기준]에 그대로 사용, 최근 달만 펼침) ③**미지급 섹션 상단 분리**(최근 일자순) ④**상세 탭에도 [전부 지급 처리]**(지급일 date 입력 = 실제 이체일 **소급 기록**, `paid_on`→`setTaskPayout/setSessionEngineerPayout` 3번째 인자; `return=detail`이면 상세 복귀) ⑤**삭제 confirm에 미지급 경고** ⑥**단가 미입력 가시화**(배정됐는데 worker_rate 0이면 참여 내역·정산 목록에 배지; `listSessionsForWorker`에 my_assigned/my_rate) — **0원도 지급·정산 가능**: 개별 [지급 처리] 버튼 상시 제공·일괄 지급/미지급 집계의 rate>0 필터 제거(0원 포함), '단가 미입력' 배지는 누락 실수 감지용 정보 표시로 유지(의도한 무료면 그대로 지급 처리 안내) ⑦상세→연동 연락처 크로스링크(party_id) ⑧정보 수정 폼 dirty 저장 패턴. **원천징수 3.3% 표시**: 목록 카드 미지급 요약 아래 `원천세 3.3% −₩X → 실지급 ₩Y` 한 줄 + 정산 탭 합계 카드에 `소득세+지방소득세` 분해 줄(미지급 기준). 계산=`lib/tax.js withholding33`(소득세 3%·지방세는 소득세의 10%, **각 10원 미만 절사 — 홈택스 신고 방식**). **표시 참고용** — 소액부징수·사업자 외주(세금계산서 발행, 원천징수 없음) 예외 미반영(화면에 명시). 일괄 [지급 처리] confirm에도 실지급 병기. **세션 참여 표시**: 상세 탭 라벨 **'참여 내역'**(작업+세션 합계 개수), `listSessionsForWorker`(다대다 `session_engineers.manager_id` 매칭 + 레거시 `engineer_name` 폴백, DISTINCT로 중복 제거) — 탭 안에 '작업 N'/'세션 N' 두 섹션으로 나눠 표시(세션은 `worker_rate`/정산 개념이 없어 참고용, 배지만·금액 없음). ## 자료 전달 아키텍처 (플레이북1 §2.3·§4.3)

- **스토리지 추상화(`src/storage.js`)**: Drive 연동 시 Drive, 미연동 시 로컬 디스크(`config.uploadsDir`).

  multipart 파일명은 latin1 → UTF-8 복원(한글 보존).
- **비공개 프록시 스트리밍**: 파일은 공개 URL 없음. 인증 다운로드 `/deliverables/:id/raw`(범위 강제) +
  공개 링크 `/d/:token`(로그인 불필요). 둘 다 백엔드가 프록시.
- **만료 토큰 링크**: 난수 `access_token`, `expires_at` 만료, `revoked` 철회, `download_count` 추적.
  게이트(`tokenGate`)가 존재/철회/만료를 검사.

26. 【📏 관례】**Tailwind display 유틸(`flex`/`grid`/`inline-block` 등)이 붙은 요소는 `hidden` 속성·`el.hidden=true`로 안 숨겨진다**(2026-07-15 사용자 리포트 '클라이언트 실시간 필터가 안 먹고 전체 목록 그대로' — 목록 실시간 필터 첫 배포에서 재발): `[hidden]{display:none}`은 UA/preflight 스타일이라 **author 스타일인 display 유틸(같은 특정도, 소스 순서상 뒤)에 밀린다** → `.flex` 행에 `hidden`을 걸어도 `display:flex`가 이겨 안 사라진다. JS로 확실히 숨기려면 **인라인 `el.style.display="none"`**(인라인=특정도 1,0,0,0으로 클래스 규칙을 항상 이김; 복원은 `el.style.display=""`). display 유틸이 없는 요소(예: 빈 결과 안내 `block`)만 `hidden`으로 충분. ⚠️**jsdom은 CSS를 적용하지 않아** `el.hidden=true`가 '통과'하지만(속성만 반영·실제 숨김 미검증) 실브라우저에선 실패 — jsdom 테스트는 `.hidden`이 아니라 `el.style.display`로 검증할 것(이 버그가 jsdom 테스트를 통과해 배포됐다).
27. 【📏 관례】**서버 렌더 인라인 `style="…"`는 CSP `style-src`에 막혀 조용히 무시된다 — 폭·레이아웃은 CSS 클래스로**(2026-07-16 사용자 리포트 '표 오른쪽 여백·이메일 잘림' — `dataTable`의 `<col style="width:…">`가 안 먹어 6열 균등 분배로 렌더되던 근본 원인): CSP가 `style-src 'self' …`(unsafe-inline 없음)라 **서버가 HTML에 박은 `style=` 속성은 브라우저가 파싱조차 안 한다**(`el.style.width`가 빈 값). 반면 app.js의 `el.style.x=…`(JS 프로퍼티 세팅)는 style-src와 무관하게 동작 → '앱은 되는데 서버 렌더만 안 되는' 혼란. 그래서 `inv-table`·`proj-thead`는 처음부터 외부 CSS(`.inv-w-*`·grid-template-columns)로 폭을 준다. **새 표·레이아웃의 치수는 반드시 CSS 클래스로**(Tailwind 임의값 `w-[16rem]` 등 — 콜러가 **리터럴**로 넘겨야 Tailwind가 스캔·생성, 동적 `w-${x}`는 안 잡힘). `<col>`은 폭만 주면 숨겨도 폭을 예약하므로, 반응형 hide 시 col도 `hidden xl:table-column`으로 함께 숨겨 예약을 없앤다(셀은 `table-cell`, 콜은 `table-column`). 참고: JS로 토글하는 `style="display:none"`(inv-bulkbar)은 로드 시 app.js가 다시 세팅해 기능은 유지되나, **초기 상태를 인라인 style에 의존하지 말 것**.
