# 사람/조직 축 정리 — 연락처 = 사람, 업체·그룹 = 조직

> 2026-07-17 · 사용자 제기("클라이언트의 관계자·아티스트 탭이 연락처와 중복 — 업체랑 그룹만 남기고 개념을 새로 정리")
> 대상: `/contacts`(연락처) · `/clients`(현 클라이언트 → 업체·그룹)

## 배경 · 문제

같은 사람을 두 메뉴, 세 목록에서 관리하고 있다. 프로덕션 실측(2026-07-17):

| 연락처(사람) | | 클라이언트 | |
|---|---|---|---|
| 외부 202 · 외주 4 · 스태프 5 | **211명** | 업체 119 · **관계자 151** · **아티스트 73** · 그룹 20 | |

- **관계자 151명 전원이 이미 연락처에 있다**(연락처에 없는 관계자 = 0).
- **아티스트 73명 전원이 이미 연락처에 있다**(연락처에 없는 아티스트 = 0).
- 클라이언트 안에서도 **22명이 관계자·아티스트 두 탭에 동시 노출**된다.

**근본 원인은 분류 축이 섞인 것이다.** 연락처는 *무엇인가*(사람)로 자르는데,
클라이언트는 *무엇인가*(업체·그룹)와 *어떤 역할인가*(관계자·아티스트)를 **한 탭바에 섞어** 자른다.
역할은 겸업 가능(아티스트이면서 디렉터)이라 배타적일 수 없고, 그래서 22명이 두 탭에 겹친다.

부수 증상:
- **사람 상세가 두 벌**: 아티스트는 `/clients/:id`(clientForm 인라인 편집)와 `/contacts/:id`(읽기 뷰) 양쪽에서 열린다.
- **폼이 두 벌**: 아티스트 폼과 연락처 폼이 같은 필드를 각각 갖는다(활동명·현금영수증·소속 그룹).
  아티스트의 '소속사'와 연락처의 '회사'는 **같은 저장소**(`affiliations` 현재 소속)를 서로 다른 경로
  (`setPartyAgency` vs `syncCompanyAffiliation`)로 쓴다.

## 결정 사항(사용자)

| # | 항목 | 결정 |
|---|---|---|
| 1 | 클라이언트의 정체성 | **조직 명부**(업체·그룹). 개인은 청구처가 되더라도 연락처에서 관리 |
| 2 | 연락처 탭 | **전체 / 아티스트 / 관계자 / 외주 / 스태프** — 겹침을 인정하는 **역할 필터**, '전체'가 기본 |
| 3 | 활동 형태(`activity_form`) | **UI 폐기**(유일한 표시처인 아티스트 탭 열이 사라짐). 컬럼·config·백필은 레거시로 잔존 |
| 4 | 소속사/회사 라벨 | **'소속'** 하나로 통합(재직·매니지먼트 둘 다 포괄, 기존 '소속 이력'과 일치) |
| 5 | 메뉴 이름 | **'업체·그룹'**(URL `/clients`·코드 식별자·DB 컬럼은 불변 — 배포 안정성) |
| 6 | '클라이언트' 표현 | **화면에서 전부 제거**(코드·DB 식별자는 유지) |

## 목표 · 비목표

**목표**
- 분류 축을 하나로: **사람 = 연락처, 조직 = 업체·그룹**. 역할은 메뉴가 아니라 **배지·필터**로만.
- 사람 상세·편집 폼을 **연락처 한 곳으로** 일원화.
- 화면 용어에서 '클라이언트' 제거.

**비목표(이번에 안 함)**
- 데이터 이동·마이그레이션 — `parties` 한 테이블이라 **필요 없다**. 순수 화면 정리다.
- URL 변경(`/clients` 유지) · 코드 식별자 리네임(`listClients`·`client_id`·`clients.routes.js` 등 유지)
- 청구처(payer) 로직 · 프로젝트 폼 콤보 · 업체 첨부 · 그룹 멤버 관리 기능 자체
- `activity_form` 컬럼 드롭(레거시 보존)

## 화면 구조

### 연락처(`/contacts`) — 사람 전부

탭 = **역할 필터**(상호배타 아님, 겹침 정상):

| 탭 | 정의 | 실측 |
|---|---|---|
| 전체(기본) | `kind='person'` | 211 |
| 아티스트 | `person` + `is_artist=1` | 73 |
| 관계자 | `listAssociates` 로직(`person`·비스태프·비외주·(`is_artist=0` OR 관계자 역할 참조)) | 151 |
| 외주 | `person`·`user_id IS NULL`·외주 작업자 참조 | 4 |
| 스태프 | `person`·`user_id IS NOT NULL` | 5 |

- 탭 라벨에 개수 표시. '전체'가 기본(현재 기본 '외부 연락처'는 폐기 — '전체'가 대체).
- 아티스트 73명은 '전체'와 '아티스트'에, 아티스트 겸 디렉터 22명은 '아티스트'와 '관계자'에 함께 나온다(설계 의도).
- 2단 마스터-디테일·이름만 목록·↑↓ 이동·읽기 뷰는 **그대로**.

### 업체·그룹(`/clients`) — 조직만

- 탭 = **업체 119 / 그룹 20**. 관계자·아티스트 탭 제거.
- `/clients?group=associate` → `/contacts?tab=associate`, `/clients?group=artist` → `/contacts?tab=artist` **리다이렉트**(북마크 호환).
- **사람 id로 `/clients/:id`를 열면 `/contacts/:id`로 리다이렉트** — 지금은 비아티스트만 그렇게 동작하므로 **아티스트로 확장**(`?from=`·`?return=` 보존).
- '새 클라이언트' 드롭다운 → **업체 / 그룹 2택**. 사람 생성은 연락처의 '+ 새 연락처'.

### 사람 편집 폼 — 연락처 하나로

`clientForm`의 `type === "artist"` 분기를 **제거**한다. 이관·통합 결과:

| 아티스트 폼 필드 | 처리 |
|---|---|
| 활동명 | 연락처 폼에 이미 있음(`nickname` → `activity_name`) |
| 현금영수증 | 연락처 폼에 이미 있음 |
| 소속 그룹 | 연락처 폼에 이미 있음(`group_id`) |
| **소속사**(`agency_company` → `setPartyAgency`) | 연락처 폼의 **'회사'와 같은 데이터**(`affiliations` 현재 소속) → **칸 하나로 통합, 라벨 '소속'**. 저장은 연락처의 기존 경로(`syncCompanyAffiliation`) |
| **활동 형태**(`activity_form`) | **폐기**(select 제거) |

`clientForm`은 **업체·그룹 전용**이 된다.

## 용어 정리(화면에서 '클라이언트' 제거)

문맥에 따라 다른 말이 정확하다 — 일괄 치환하면 안 된다:

| 화면 문맥 | 지금 | 바꿀 말 | 근거 |
|---|---|---|---|
| 사이드바 메뉴·페이지 제목 | 클라이언트 | **업체·그룹** | 조직 명부 |
| 백링크 라벨(`safePath` 판정) | 클라이언트 | **업체·그룹** | 〃 |
| 개별 대상 지칭(수정·삭제·오류) | 클라이언트 수정/삭제/찾을 수 없음 | **업체**/**그룹** 수정·삭제(유형별) | 대상이 둘 중 하나로 확정됨 |
| 연락처 읽기 뷰 파생정보 | 대표 클라이언트 | **대표 업체** | 대표자는 업체의 대표 |
| **청구 맥락**(청구 목록 열·검색 placeholder·발행 오류·청구처 카드 링크) | 클라이언트 | **청구처** | **개인도 청구처가 된다** — '업체'로 바꾸면 틀림 |
| 첨부·자료 등 기타 | 클라이언트 | 문맥에 맞는 말(업체/청구처) | — |

**불변**: URL(`/clients`)·라우트 파일명·함수명(`listClients`·`clientForm`)·DB 컬럼(`client_id`·`client_files.client_id`)·
내부 주석. 사용자에게 보이는 **문자열 리터럴만** 바꾼다.

## 오류 · 예외

- `/contacts?tab=<모르는 값>` → '전체'로 폴백(현행 탭 화이트리스트 패턴).
- `/clients?group=associate|artist` → 대응 연락처 탭으로 **302**(검색어 `q`가 있으면 함께 전달).
- 사람 id로 `/clients/:id` → `/contacts/:id` 302(`from`·`return` 보존).
- 조직 id로 `/contacts/:id` → 현행처럼 404(연락처는 사람만).

## 테스트

**신규**
- `listContacts({tab})` 5분기 계약: 전체 ⊇ 아티스트·관계자·외주·스태프, 아티스트∩관계자 겹침 허용, 스태프·외주 배타.
- 리다이렉트: `/clients?group=artist`→`/contacts?tab=artist`, `?group=associate`→`?tab=associate`(q 보존), 아티스트 id `/clients/:id`→`/contacts/:id`.
- 연락처 폼: '소속' 칸 저장 시 `affiliations` 현재 소속 갱신(아티스트·관계자 동일 경로), 활동 형태 필드 부재.
- 업체·그룹 목록: 탭 2개만, '새 클라이언트' 드롭다운에 사람 유형 없음.

**갱신**
- 클라이언트 관계자 2단 테스트(`contacts-panes`) 제거, 아티스트 탭 열 테스트 제거.
- 용어 계약: 화면 문자열에 '클라이언트'가 없는지 검사하는 가드(허용 예외 = 코드 식별자·주석). **반복 실수 클래스라 기계 가드로 승격**(CLAUDE.md 함정 #21 정책).

**E2E(실브라우저)**
- 연락처 5탭 개수(211/73/151/4/5)와 겹침이 실측치와 일치.
- 아티스트를 연락처에서 열어 '소속' 수정 → 업체 상세 '소속 아티스트'에 반영.
- 옛 링크(`/clients?group=artist`·아티스트 `/clients/:id`) 진입 시 연락처로 안착.

## 위험

| 위험 | 완화 |
|---|---|
| 아티스트 편집 경로가 바뀜(클라이언트 상세 → 연락처) | 리다이렉트로 옛 링크 흡수. 필드는 그대로(활동 형태만 폐기) |
| '소속' 통합 시 저장 경로가 `setPartyAgency`→`syncCompanyAffiliation`으로 바뀜 | 둘 다 `affiliations` 현재 소속을 갱신한다. **구현 첫 태스크에서 두 경로의 결과 동등성을 테스트로 고정**한 뒤 통합할 것(같은 회사명 입력 시 같은 현재 소속 1건). 또한 **그룹 선택 시 소속사 자동 채움**(app.js `fillAgency`)이 연락처 폼에서도 동작해야 한다 — 지금 연락처 폼엔 소속사 콤보가 없어 no-op이므로, '소속' 칸 도입과 함께 jsdom 테스트로 확인 |
| 용어 일괄 치환의 과잉(청구 맥락까지 '업체'로) | 위 매핑 표대로 문맥별 치환. 청구 맥락은 '청구처' |
| 오늘 만든 관계자 2단 폐기 | 렌더러는 연락처에서 계속 쓰인다 — 버리는 건 클라이언트 쪽 배선뿐 |
| 활동 형태 폐기로 정보 손실 | 컬럼·백필 보존이라 되살릴 수 있음. 표시처가 없어 실질 손실 없음 |
